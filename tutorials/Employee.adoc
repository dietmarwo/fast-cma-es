== Employee Scheduling

This tutorial:

- Is based on
https://github.com/optapy/optapy-quickstarts/blob/stable/employee-scheduling/README.adoc[employee-scheduling],
an https://github.com/optapy/optapy[OptaPy] tutorial for employee scheduling.
- Shows how to improve the OptaPy result by applying parallel continuous optimization.
- Illustrates how easy it is to define the corresponding fitness function.
- Applies multi-objective optimization to generate a set of non-dominated scheduling choices
with an additional competing objective.

The code for this tutorial is
here:

- https://github.com/dietmarwo/fast-cma-es/blob/master/examples/employee.py[employee.py].

== Motivation

Employee scheduling is a common optimization problem. It includes hard constraints,
for example when an employee lacks the required skill for a slot or is unavailable on a given day.
It also includes soft constraints, for example when an employee prefers a specific shift.
These constraints need clear priorities.

Continuous optimization is not a common first choice here. Recent developments make it far more practical.
The main reasons are:

- Using https://numba.pydata.org/[numba] and parallel execution on a modern many-core CPU, you can execute up to
1000.000 evaluations/sec of a fitness function that evaluates a possible schedule.
- Modern continuous optimization algorithms like https://github.com/avaneev/biteopt[BiteOpt] can handle discrete problems very well.

This means that even if a scheduling or planning problem needs millions of fitness evaluations,
the runtime can still be measured in seconds.

That is only part of the story. What if you want to add another competing objective and generate
several non-dominated choices, a so-called pareto-front? In continuous optimization this is largely a solved problem.
You get that capability almost for free. Traditional scheduling optimizers such as
https://github.com/optapy/optapy[OptaPy] do not make this as straightforward.

One possible objection is that a dedicated scheduling API may be easier to use for this kind of problem.
A fair answer is to show an alternative. We will define a fitness function for the same problem
and let the reader decide which approach is easier.

=== Representing the Employee Scheduling JSON instance as Optimization Problem

We can now read the solution-independent parts of the generated JSON and turn them into
a continuous optimization problem with a fitness function:

[source,python]
----
class problem():

    def __init__(self, json_file):
        with open(json_file) as json_file:
            sched = json.load(json_file)

        self.shifts = sched['shift_list']
        self.shift_to_index, self.days, self.locations, self.required_skills, \
                self.sec_start, self.sec_end = shift_indices(self.shifts)
        self.day_to_index, self.day_ids = index_map(self.days)
        self.location_to_index, self.location_ids = index_map(self.locations)

        self.employees = sched['employee_list']
        self.employee_to_index, self.names, self.skill_sets = employee_indices(self.employees)
        self.name_to_index, self.name_ids = index_map(self.names)
        self.skill_to_index, self.skill_set_ids = index_multi_map(self.skill_sets)
        self.required_skill_ids = np.array([self.skill_to_index[s] for s in self.required_skills])

        self.avails = sched['availability_list']
        self.avail_to_index, self.avail_names, self.avail_types, self.avail_days = avail_indices(self.avails)
        self.avail_name_ids = np.array([self.name_to_index[n] for n in self.avail_names])
        self.avail_day_ids = np.array([self.day_to_index[d] for d in self.avail_days])
        self.avail_type_ids = np.array([avail_type_map[t] for t in self.avail_types])

        self.dim = len(self.shifts)
        self.bounds = Bounds([0]*self.dim, [len(self.employees)-1E-9]*self.dim)

    def fitness(self, x):
        score, employee_num_shifts = fitness_(x.astype(int), self.day_ids,
                    self.required_skill_ids, self.skill_set_ids, self.avail_name_ids,
                    self.avail_day_ids, self.avail_type_ids, self.sec_start, self.sec_end)
        return score + 10*np.std(employee_num_shifts)
----

We convert all information into numpy index arrays, together with lists that let us recover
the original representation from these indices.

This representation speeds up fitness evaluation when using https://numba.pydata.org/[numba].
numba works well with numpy arrays and poorly with Python objects.
The indexed form also makes comparisons faster.

The fitness function forwards these index arrays to the fast numba function
`fitness_(x.astype(int), ...` discussed below. Note that the continuous decision vector
`x` is converted to discrete integer values using `x.astype(int)`.

=== Implementing the Fitness Function

The fitness function checks how many hard and soft constraints
an employee schedule `employees_at_shift`, given as a decision vector, violates.
We multiply hard constraints by 1000 to prioritize them. `UNDESIRED`
constraints, where an employee prefers not to work on a specific day,
get factor 100. `DESIRED` constraints,
where an employee wants to work on a specific day, get a negative
factor `-1`, because we want to maximize their fulfillment.

[source,python]
----
@njit(fastmath=True)
def fitness_(employees_at_shift, day_ids, required_skill_ids, skill_set_ids,
             avail_names_ids, avail_days_ids, avail_type_ids, sec_start, sec_end):
    score = 0
    num_employees = len(skill_set_ids)
    employee_last_day = np.full(num_employees, -1, dtype=numba.int32)
    employee_last_end = np.full(num_employees, -1, dtype=numba.int32)
    employee_num_shifts = np.zeros(num_employees, dtype=numba.int32)
    for shift in range(len(employees_at_shift)):
        day = day_ids[shift]
        employee = employees_at_shift[shift]
        employee_num_shifts[employee] += 1
        if employee_last_day[employee] == day:
            score += 1000  # employee should only work once a day
        employee_last_day[employee] = day
        if sec_start[shift] - employee_last_end[employee] < 10*3600:
            score += 1000  # employee should pause for 10 hours (and shifts should not overlap)
        employee_last_end[employee] = sec_end[shift]
        required_skill = required_skill_ids[shift]
        skill_set = skill_set_ids[employee]
        if not required_skill in skill_set:
            score += 1000 # employee has wrong skill set
        avail_ids = np.where(avail_names_ids == employee)
        for avail_id in avail_ids[0]:
            avail_day = avail_days_ids[avail_id]
            if day == avail_day:
                type = avail_type_ids[avail_id]
                if type == UNDESIRED:
                    score += 100 # employee does not want to work this day
                elif type == UNAVAILABLE:
                    score += 1000 # employee is unavailable
                elif type == DESIRED:
                    score -= 100 # employee works at desired day
    return score, employee_num_shifts
----

You may compare the complexity of this code to
https://github.com/optapy/optapy-quickstarts/blob/stable/employee-scheduling/constraints.py[constraints.py] and
https://github.com/optapy/optapy-quickstarts/blob/stable/employee-scheduling/domain.py[domain.py].
The fitness function above does not require specific domain objects or a
"schedule solver" API, but it is still quite readable. It also does more:
it counts the number of shifts for each employee and returns this as an array. You can use
`np.std(employee_num_shifts)` or `-min(employee_num_shifts)` to support a more
equal distribution of work. This is how we encourage a fair number of shifts for all employees.

=== Single Objective Optimization

We call `fcmaes.retry.minimize_plot` because we want to monitor and plot progress over time.
It takes a continuous optimizer as an argument. We recommend trying BiteOpt first, not only for
this problem, because it does not need special parameter tuning and is mostly self-adapting.
`fcmaes.retry` uses `mp.cpu_count()` parallel workers by default. On our AMD 16 core 5950x,
this leads to 32 optimizations running in parallel.

[source,python]
----
    def fitness(self, x):
        score, employee_num_shifts = fitness_(x.astype(int), self.day_ids,
                    self.required_skill_ids, self.skill_set_ids, self.avail_name_ids,
                    self.avail_day_ids, self.avail_type_ids, self.sec_start, self.sec_end)
        return score + 10*np.std(employee_num_shifts)

    def optimize(self):
        self.fitness(np.random.uniform(0, len(self.employees), self.dim).astype(int))
        res = retry.minimize_plot("schedule.bite.400k", Bite_cpp(400000),
        #res = retry.minimize_plot("schedule.de.400k", De_cpp(400000, popsize = 512, ints = [True]*self.dim),
                    wrapper(self.fitness), self.bounds, num_retries=32, plot_limit=10000)
        print(self.fitness_mo(res.x))
        self.show(res.x)

=== Multi-Objective Fitness

For the fitness function, the only change is that instead of adding `-0.1*np.std(employee_num_shifts)` to the first objective,
we return `np.std(employee_num_shifts)` as a second objective. Alternatively, we could maximize
the minimum number of assigned shifts for an employee with `-min(employee_num_shifts)`.

[source,python]
----
    def fitness_mo(self, x):
        score, employee_num_shifts = fitness_(x.astype(int), self.day_ids,
                    self.required_skill_ids, self.skill_set_ids, self.avail_name_ids,
                    self.avail_day_ids, self.avail_type_ids, self.sec_start, self.sec_end)
        return [score, np.std(employee_num_shifts)]
        #return [score, -min(employee_num_shifts)]
----

=== Multi-Objective Optimization

Since `fcmaes` offers only one multi-objective optimizer, `MODE`, the main
choice is whether to use differential evolution or NSGA-II population
update through the parameter `nsga_update=True`. It is worth trying both.
For this problem, NSGA-II population update works better.
Multi-objective optimization usually needs a larger population size, so we choose 512 here.

[source,python]
----
    def optimize_mo(self):
        self.fitness_mo(np.random.uniform(0, len(self.employees), self.dim).astype(int))
        pname = "schedule_mo_600k.512"
        xs, ys = modecpp.retry(mode.wrapper(self.fitness_mo, 2),
                     2, 0, self.bounds, popsize = 512, max_evaluations = 600000,
                     nsga_update=True, num_retries = 32, workers=32)
        np.savez_compressed(pname, xs=xs, ys=ys)
        xs, ys = moretry.pareto(xs, ys)
        for x, y in zip(xs, ys):
            print(str(list(y)) + ' ' + str([int(xi) for xi in x]))
----

    def optimize_ng(self):
        import nevergrad as ng
        fit = wrapper(self.fitness)
        instrum = ng.p.Instrumentation(
            ng.p.Array(shape=(self.dim,)).set_bounds(self.bounds.lb, self.bounds.ub)
            )
        optimizer = ng.optimizers.TwoPointsDE(parametrization=instrum, budget=100000)
        recommendation = optimizer.minimize(fit)
        print(recommendation.value)
        self.show(recommendation.value[0][0])

== Conclusion

- Multi-objective optimization can support decision-making even for large combinatorial scheduling problems.
- It can help make employees "happy" by producing schedules that fulfill all their "desires" and avoid leaving some employees without assigned shifts.
- Performance of continuous optimization is sufficient even for large problem instances in Python if https://numba.pydata.org/[numba] is used for the fitness function.
- Modern continuous optimizers written in {cpp}, like BiteOpt and fcmaes-MODE, enable up to 10⁶ fitness evaluations/sec and work well when decision variables are used as discrete integer values.
- The approach shown here is flexible and handles unusual constraints and modifications well. For example, you could assign different weights to soft constraints for individual employees.
- Standard tools reach their limits quickly. Continuous optimization can sometimes still fulfill all hard and soft constraints, although it may require many millions of fitness evaluations.
- For problems where the requirements are very hard to fulfill, the pareto-front generated by multi-objective optimization can be very small, but it can still offer interesting alternatives.
